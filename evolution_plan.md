# Evolution plan: Synchatica → «движок ИИ‑персон» уровня рынка без аналогов

## 1) Контекст и текущая архитектура (статический обзор)

**Точки входа и оркестрация**
- `main.py` запускает API и/или Telegram‑бота, включает планировщик и аккуратно закрывает ресурсы (Redis, HTTP‑клиент, БД).【F:main.py†L1-L152】

**API слой**
- FastAPI приложение с маршрутом `/api/v1` для диалогов и `/healthz` для health‑checks.【F:app/api/app.py†L1-L17】
- Валидация входных данных (текст/изображение/голос), лимиты, MIME‑проверки и JSON schema‑валидация лежат в `app/api/conversation.py`.【F:app/api/conversation.py†L1-L200】

**Telegram‑бот**
- Вебхук‑сервер на aiohttp, интеграция с aiogram, idempotency через Redis, автозагрузка локализованных приветствий.【F:app/bot/components/webhook.py†L1-L170】

**Персона/эмо‑ядро**
- Класс `Persona` управляет эмоциональным состоянием, модулями стиля, фоном, обновлением метрик и интеграцией с памятью/мозгом персонажа.【F:app/emo_engine/persona/core.py†L1-L200】
- Подсистема памяти использует Redis + RedisSearch (векторные индексы), ранжирование по сходству/салиентности/давности и механизмы консолидации/дедупликации.【F:app/emo_engine/persona/memory.py†L1-L200】

**Данные и инфраструктура**
- Async SQLAlchemy с тонкой настройкой пула/таймаутов, включая read‑only сессии и логирование «медленных» SQL при включении флага.【F:app/core/db.py†L1-L116】
- Redis‑клиент c retry, метрикой медленных операций и безопасным логированием аргументов.【F:app/core/memory.py†L1-L200】

**Планировщик задач**
- APScheduler управляет периодическими задачами (посты, аналитика, чистки).【F:app/tasks/scheduler.py†L1-L200】

**Интеграции LLM**
- Клиент OpenAI с контролем параллелизма, retry, таймаутами и нормализацией параметров запросов.【F:app/clients/openai_client.py†L1-L200】

**Конфигурация**
- Централизованные env‑настройки для API/бота/LLM/хранилищ/лимитов/безопасности через `app/config.py`.【F:app/config.py†L1-L200】

## 2) Сильные стороны текущей базы
- **Модульность**: выделены слои API, бота, ядра персоны, хранилищ и задач. Это упрощает поэтапную эволюцию.【F:main.py†L1-L152】【F:app/api/app.py†L1-L17】【F:app/bot/components/webhook.py†L1-L170】
- **Богатая память**: продвинутый memory‑pipeline с векторным поиском, ранжированием, дедупликацией и параметрами влияния эмоций/давности.【F:app/emo_engine/persona/memory.py†L1-L200】
- **Контроль стабильности**: retry на Redis/LLM, таймауты БД и медленных операций, защита от перегрузок.【F:app/core/memory.py†L1-L200】【F:app/clients/openai_client.py†L1-L200】【F:app/core/db.py†L1-L116】

## 3) Ключевые пробелы и риски (на основе кода)
1. **Ограниченная наблюдаемость** — нет явной системы метрик/трейсинга по «персональным» конвейерам (эмо‑модель, память, LLM‑вызовы). Сейчас есть логирование, но нет единой телеметрии. (Нужно поверх существующих логов/таймаутов.)【F:app/core/memory.py†L1-L200】【F:app/clients/openai_client.py†L1-L200】
2. **Слабая формализация контрактов персоны** — поведение эмо‑ядра и памяти трудно тестировать, поскольку нет выделенного API‑контракта/контекст‑объекта для эволюции логики. (Видно в «сильной» связанности `Persona`.)【F:app/emo_engine/persona/core.py†L1-L200】
3. **Оценка качества и регрессии** — в репозитории есть только один тест планировщика; нет измеримых критериев качества персон/памяти/стиля/безопасности.【F:tests/test_scheduler.py†L1-L200】
4. **Сложность масштабирования** — Redis/DB/LLM интеграции настроены на стабильность, но нет видимых изолированных очередей/пайплайнов для масштабирования «персон» по нагрузке. (Элементы есть, но единый масштабный pipeline не формализован.)【F:app/core/memory.py†L1-L200】【F:app/emo_engine/persona/memory.py†L1-L200】
5. **Прозрачность конфигурации** — множество env‑параметров без формализованных профилей (dev/stage/prod) и без валидации взаимоисключающих флагов. Это усложняет управляемость. 【F:app/config.py†L1-L200】

## 4) Целевой образ продукта (North Star)
**Synchatica = движок ИИ‑персон**, где персона — это управляемый объект с:
- богатой эпизодической/семантической памятью;
- эмоциональной динамикой, адаптирующейся к контексту и пользователю;
- механизмом устойчивого долгосрочного развития личности;
- управляемыми стилями/ролями и безопасными ограничениями;
- формализованной «качественной метрикой» (оценка полезности, естественности, последовательности).

## 5) Эволюционный план (по фазам)

### Фаза 0 — Статус‑кво + «безопасная основа» (0–4 недели)
**Цели:** повысить управляемость, наблюдаемость, тестируемость без изменения внешнего поведения.
1) **Документирование текущих потоков**
   - краткие схемы: «API → LLM → Memory → Response», «Telegram → Persona → Storage». 
2) **Минимальная телеметрия**
   - единый формат логов (request_id, chat_id, persona_id, latency breakdown);
   - метрики latency/ошибок/событий (LLM, Redis, DB, pipeline). 
3) **Базовый quality‑harness**
   - стабильные тест‑кейсы персон (FAQ‑диалоги, память, стиль);
   - фиксация baseline‑ответов и метрик (semantic similarity, safety flags).
4) **Валидация конфигурации**
   - схемы/проверки для ключевых env‑параметров (особенно API/LLM/Redis).

**Выход фазы:** наблюдаемая и измеримая система, в которой регрессии можно поймать до деплоя.

### Фаза 1 — Персона как «контракт» (1–3 месяца)
**Цели:** сделать персону управляемым и расширяемым объектом.
1) **Стабилизация API контекста персоны**
   - выделить «PersonaContext» (identity + preferences + memory handles) для уменьшения связанности; 
   - формализация входов/выходов ключевых частей (emotion update, memory write, response synthesis).
2) **Память v2**
   - расширение стратегии консолидации (episodic → semantic);
   - атрибуты доверия/достоверности и «давности правды»; 
   - контроль drift (неперезаписываемые факты vs гипотезы).
3) **Стилевые режимы**
   - явная модель «стиля» (тональность/энергия/скорость), связь с эмоциональной динамикой;
   - хранение «профилей» для разных пользователей/сценариев.

**Выход фазы:** персона с четкими контрактами и воспроизводимым поведением, готовая к масштабированию и A/B‑экспериментам.

### Фаза 2 — «Мозг персоны» и диалоговый интеллект (3–6 месяцев)
**Цели:** усилить уникальность и глубину персонажа.
1) **Многоуровневое рассуждение**
   - раздельные «thought‑процессы» для планирования ответа и стилевого оформления;
   - адаптивные budgets по токенам/latency.
2) **Расширенная эмо‑модель**
   - обучение/калибровка параметров эмо‑динамики на пользовательских данных;
   - устойчивость к «эмо‑шуму» (спам/троллинг/неустойчивый контекст).
3) **Смысловая устойчивость**
   - механизм self‑consistency (проверка на противоречия с памятью);
   - «semantic guardrails» (доверие к источникам, safety policies).

**Выход фазы:** персона «запоминает, рассуждает и развивается» последовательно и на уровне, недоступном конкурентам.

### Фаза 3 — Платформа (6–12 месяцев)
**Цели:** превратить проект в платформу для множества персон.
1) **Мульти‑персонная архитектура**
   - многопрофильные персонажи, переключение «ролей» без потери контекста;
   - marketplace персон (портирование конфигов/стилей).
2) **Инструменты для создателей персон**
   - UI/CLI для создания персона‑профилей;
   - тест‑батареи, стресс‑тесты и отчетность.
3) **Масштабирование и SRE**
   - sharding памяти и очередей;
   - оптимизация RedisSearch + кэш политики;
   - SLA по latency и resiliency.

**Выход фазы:** рынок персон и устойчивый бизнес‑уровень надежности.

## 6) Метрики успеха
- **Стабильность**: P95/P99 latency per request; Error rate < 0.5%.
- **Персонализация**: рост долгосрочной вовлеченности, repeat‑sessions, retention.
- **Качество**: тест‑пакеты на consistency/memory recall/style adherence.
- **Безопасность**: rate‑limit, moderation coverage, minimal unsafe responses.

## 7) Риски и способы снижения
- **Регрессии поведения персоны** → baseline‑test suite + A/B‑эксперименты. 
- **Рост задержек** → профилирование LLM/Redis/DB + кэширование + ограничение контекста.
- **Сложность разработки** → формализация контрактов и качественных метрик.
